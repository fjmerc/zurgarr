## [Unreleased]

### Fixed

- **Library symlink cleanup: broken debrid symlinks now trigger Sonarr/Radarr re-search instead of disappearing silently**: `library.py:_cleanup_broken_debrid_symlinks` runs every library scan and deletes symlinks whose debrid targets no longer exist on the rclone mount (e.g., when a debrid release is removed from the user's RD library). Until now it deleted the symlinks without telling the arrs — Sonarr/Radarr's database still held file IDs pointing at the deleted paths, so they continued reporting the affected episodes/movies as "downloaded" until their next disk scan. The 6-hourly `verify_symlinks` task does trigger a re-search via `_attempt_arr_research` (phase 3), but the library scan runs more often and beat it to the deletion, so by the time `verify_symlinks` fired it found 0 broken symlinks and triggered 0 re-searches. Real-world impact: a user whose `Dexter S01-S08 FraMeSToR [RiCK]` pack was removed from RD lost all 12 S01 + 12 S05 .mkv symlinks (no separate per-season releases existed for those two seasons), but Sonarr still showed both seasons as "Available" because nothing told it to re-stat. The Library page correctly reported `episodes=0` for those seasons because the broken `.mkv`s no longer counted as media files; Sonarr disagreed because its DB was stale. Fix: `_cleanup_broken_debrid_symlinks` now records the unique release names of every deleted symlink (extracted via `scheduled_tasks._extract_release_info` from the symlink target — same parser `verify_symlinks` already uses), and after the deletion loop, gated on `gap_fill_enabled()` (default `true`, mirrors `verify_symlinks`' opt-out), iterates the deduped set and calls `scheduled_tasks._attempt_arr_research(release_name)` for each. That helper parses the release name, looks up the show/movie in Sonarr/Radarr by title, and triggers a search for episodes/movies that lack files — with its existing `_retrigger_history` cooldown so frequent library scans (default 30 min) can't stampede searches for the same release. Sonarr/Radarr parity is preserved because `_attempt_arr_research` already routes by `is_tv` from the parsed release name. The function now also emits a `log_event('cleanup', 'Library Symlink Cleanup', meta={'cause': 'library_symlink_cleanup', 'deleted': N, 'searched': M})` so the work is visible in the Activity feed instead of only the per-symlink debug log line; new `library_symlink_cleanup` cause registered in `history.py` plus matching Python and JS formatters in `activity_format.py` per the CLAUDE.md "add a new cause in three places" rule. Lazy imports (`from utils.scheduled_tasks import ...` inside the function) avoid a circular import since `scheduled_tasks` already imports `library.gap_fill_enabled` at the top of `verify_symlinks`. Cleanup-without-deletions short-circuits to `return` before logging an event, so noiseless scans don't create empty Activity rows.

- **Library page: parser-junk debrid titles now merge with the canonical local entry into a single library card with `source='both'`, instead of surfacing two separate posters**: The prior commit fixed the symlink-creation cascade in `_create_debrid_symlinks` so new debrid symlinks land in the canonical Radarr/Sonarr folder. But the user still saw two posters for the same movie because the matching cascade exists in two *more* places upstream of symlink creation: (1) the title-level merge step inside `_scan_read` (lines 1590-1681) where debrid + local lists get combined into the unified library data model with a `source='both'` flag when both sides have the title, and (2) the TMDB-cache enrichment step in `_enrich_with_tmdb_cache` (lines 614-647) where each item's display title gets renamed from the parsed-folder name to the canonical TMDB title. Both used direct `_normalize_title(parsed)` lookups against `local_movie_keys` / `cached`, so a parser-junk title like `"Gattaca Ethan Hawke Sci Fi"` failed to bridge to the canonical `"Gattaca"` cache entry → no merge → no rename → two cards on the Library page even though the underlying release was for the same movie. Both cascade sites now use the same `_find_canonical_tmdb_via_prefix` helper deployed in the prior commit. Merge step: when direct + TMDB-id-alias matches both miss, run the prefix resolver against the local key set; if `_normalize_title(canonical['title'])` is in `local_movie_keys`, treat that as the matched key and register the alias bidirectionally in `alias_norms_local`. Symmetric for shows via `local_show_map`. Enrichment step: when `get_cached_posters` returns no info for the parsed key, run the prefix resolver and re-query `get_cached_posters` with the canonical title — `_maybe_rename` then fires and the item's display title becomes the canonical TMDB title. Together these mean: parsed `"Gattaca Ethan Hawke Sci Fi"` (debrid folder) + `"Gattaca"` (local folder) → ONE merged item with `source='both'`, display title `"Gattaca"`, ready for the user's `prefer-debrid` preference auto-enforcement to replace the local file with a symlink in place. The merge step pre-loads the full TMDB cache once per scan and threads it through to the enrichment step via the new `_shared_tmdb_cache=` kwarg, so both phases see a consistent snapshot — closes a transient-duplicate window where a `populate_cache` write between two loads could leave an item un-merged at merge-time then renamed at enrichment-time. **Hardening surfaced in review and fixed in the same change:** (a) The prefix-match path in the merge step had a self-loop alias bug — when the resolver returned a canonical title whose normalized form equaled the parsed key (degenerate case where both sides were already canonical), `alias_norms_local.setdefault(key, set()).add(canon_key)` would register `{"gattaca": {"gattaca"}}`, polluting downstream `len(aliases) > 0` checks and risking redundant lookup loops. The seed-loop convention at lines 1638-1646 explicitly excludes `name` from its own alias set via `group - {name}`; the new path now matches that convention via a `canon_key != key` guard. Same fix applied symmetrically to the shows merge. (b) The new prefix-match registration was a duplicate of the existing alias-match registration immediately below it — both `setdefault(...).add(...)` paths fired on the same `(key, matched_key)` pair when prefix-match hit. Set semantics made it idempotent but it was a structural seam. Removed the duplicate so registration happens exactly once at the unified `if matched_key != key:` site for both alias-match and prefix-match paths. (c) The enrichment helper `_resolve_canonical_info`'s `get_cached_posters` re-query passes the parsed item's year, which can diverge from the canonical TMDB release year (off-by-one is common for festival vs wide-release dates). The yearless fallback at the end of the helper recovers this because `get_cached_posters` stores info under both year-qualified and yearless keys (`tmdb.py:664-666`); added an inline comment documenting that the yearless fallback is the true success path when years diverge. **9 new tests** in `tests/test_library.py:TestMergeStepCanonicalFallback` (5 cases) and `TestEnrichmentCanonicalFallback` (4 cases) cover: parser-junk movie merges with local via canonical, parser-junk show merges with local via canonical, no-match-when-canonical-not-in-local (regression guard), self-loop guard prevents degenerate alias registration (the reviewer-surfaced bug), existing direct-match still works (regression guard), enrichment renames movie via prefix resolver, enrichment renames show via prefix resolver, no rename when resolver misses, direct lookup short-circuits prefix resolver. Pre-existing `TestWebDAVUnsupportedMemoization` tests that monkeypatched `_enrich_with_tmdb_cache` with a 2-arg lambda were updated to accept the new `_shared_tmdb_cache=` kwarg via `**kw`. Full suite: 1845 tests pass.

- **Library scanner: debrid releases with junk-bearing folder names now merge with the canonical local entry instead of creating an orphan folder**: When the user switched a movie from local to debrid, the library scanner could create a fresh `/local_media/movies/Gattaca Ethan Hawke Sci Fi (1997)/` directory containing only a debrid symlink, alongside the existing canonical `/local_media/movies/Gattaca (1997)/` — visible in the Library page as two separate posters for the same movie. Root cause: the matching cascade in `_create_debrid_symlinks` (movie/show dir selection + post-symlink rescan trigger, four sites total) used `cached_tmdb_movies.get(_normalize_title(parsed_title))` for its TMDB-direct fallback, which is an exact key match. For a parsed title with appended actor names + genre tags (`"Gattaca Ethan Hawke Sci Fi"`), the normalized key `"gattaca ethan hawke sci fi"` doesn't match the cache's actual key `"gattaca"` → cascade fell through to the parsed-title folder-name fallback → orphan folder. New helper `_find_canonical_tmdb_via_prefix(parsed_title, parsed_year, is_tv)` in `library.py` performs a token-aligned prefix scan over the cache: tokens of the cache key must be a strict prefix of the parsed-title's `_norm_for_matching` tokens, with the same hardening already deployed for the activity-feed resolver in `blackhole.py` — single-token candidates fail-closed (require year confirmation; defends against cache entries titled "The"/"It" prefix-matching every release), multi-token candidates fail-open on missing entry-year (legacy entries lack `release_date`/`first_air_date`), longest-prefix wins, per-entry try/except so one malformed cache row can't poison the whole scan, isinstance guards on `title`/`cache_key`/date strings to prevent AttributeError from non-string fields. Wired into all four cascade sites symmetrically per the CLAUDE.md "title matching cascade appears in 4 places that must stay symmetric" rule. With this in place, the user's Gattaca scenario now resolves: parsed `"Gattaca Ethan Hawke Sci Fi"` + year 1997 → token prefix matches cache key `"gattaca (1997)"` → canonical `tmdb_id=782` → `radarr_by_tmdb` lookup hits the existing Radarr movie → symlink lands inside the canonical `Gattaca (1997)` folder, no orphan created. **Hardening surfaced in review and fixed in the same change:** (a) The new helper was originally re-loading `/config/tmdb_cache.json` from disk on every invocation and re-acquiring `_tmdb._cache_lock` per call — multiplied by 4 cascade sites × N symlinked items, that was hundreds of redundant reads per scan AND widened the lock window for concurrent TMDB writers. The cache is now loaded once per `_create_debrid_symlinks` invocation (alongside the existing `cached_tmdb_ids` load) and threaded into all four call sites via the helper's existing `_tmdb_cache=` parameter — single read, single lock acquisition, no behavior change. (b) `_extract_tmdb_entry_year` had a dead `try/except ValueError` around `int(date[:4])` that could never fire because the preceding `date[:4].isdigit()` guard already excludes non-numeric input — removed for clarity. (c) Added a one-line comment at the movie rescan-trigger cascade explaining why the season-aware `find_show_tmdb_id_by_season` sub-step is absent there (it's TV-only and the absence is intentional symmetry-with-asymmetry, not an oversight). The duplication with `blackhole.py`'s parallel `_lookup_canonical_in_tmdb` (which returns just the canonical title string for activity-feed display) is conscious — different return-type API for different callers, lifting to a shared module would require changing the already-deployed blackhole code path and risking its 36 passing tests for zero user-visible gain. **21 new tests** in `tests/test_library.py:TestExtractTmdbEntryYear` (5 cases: release_date / first_air_date / precedence / missing / 6 malformed-input variants) and `TestFindCanonicalTmdbViaPrefix` (16 cases: actor+genre regression / year mismatch / longest-prefix tiebreaker / mid-string match rejection / single-token year-required guard in 3 sub-cases / single-token year-match-accepted / single-token no-entry-year fail-closed / multi-token entry-year fail-open / shows vs movies section routing / empty cache / empty title / non-dict entries / non-string cache keys / non-string title fields / missing tmdb_id excluded / parsed-year-None multi-token fail-open). Full suite: 1836 tests pass.

- **Activity feed: blackhole grab events now show the canonical media title instead of the raw release name**: When a `.torrent`/`.magnet` landed in the blackhole watch folder, `_enrich_for_history` populated `media_title` via `parse_release_name` — a ~30-line naive parser whose movie-path year regex requires `[.\s](\d{4})[.\s]` (literal `.` or whitespace on both sides). For a real release like `Gattaca.Ethan.Hawke.Sci.Fi.(1997).1080p.BluRay.x264-GROUP.torrent` the year is wrapped in parens, the regex misses, and the parser falls through to the quality regex — leaving `Gattaca Ethan Hawke Sci Fi (1997)` as the activity-feed title. Same failure mode for `[YYYY]` brackets, `-1997-` dashes, and any release that injects actor names or genre tags before the year. Three-step cascade in `_enrich_for_history` replaces the single naive call: (1) the existing `parse_release_name` still establishes `is_tv`/season/episode for `ep_str` (its other call sites are unchanged), (2) a new `_resolve_canonical_title` runs `library.parse_folder_name` — the ~250-line robust parser the library scanner already uses — which handles bracketed/parenthesized years (`_BRACKET_YEAR_PATTERN`, `_MID_YEAR_PATTERN`), dash-separated quality markers (`_QUALITY_PATTERN` accepts `[\s.\-_(\[]`), edition tags, language+codec tails, indexer URL prefixes, genre suffixes between title and year, and trailing bare years; (3) when the parser still leaves trailing junk (actor names + genre tag injected before the year, where there's no algorithmic way to know "Ethan Hawke" isn't part of the title), a new `_lookup_canonical_in_tmdb` does a token-aligned prefix match against the local TMDB cache (`/config/tmdb_cache.json`) — for the Gattaca filename, `["gattaca","ethan","hawke","sci","fi"]` starts with cache key `["gattaca"]` whose entry has `release_date=1997-10-24`, so the parsed year (1997) confirms and the canonical TMDB `title` field ("Gattaca") wins. Each cascade step falls back safely: TMDB module import failure, missing cache file, malformed cache entries (`isinstance(entry, dict)` check), `parse_folder_name` exceptions, year mismatch, and empty parsed title all flow back to the previous step's output, with the naive parser as the final fallback so the function never returns worse output than before. Year confirmation is fail-open when the cache entry lacks `release_date`/`first_air_date` (legacy entries). Longest-prefix-wins disambiguates "The Dark Knight" from a hypothetical "The Dark" cache entry. is_tv routes the lookup to the `shows` vs `movies` cache section so a Breaking Bad episode can't accidentally resolve through a movie spoof entry. `library.py` already exposes the public aliases `parse_folder_name`/`normalize_title`/`norm_for_matching` for cross-module reuse, and the `tmdb` import is lazy inside the resolver so module-load order and circular-import risk are unchanged. **Hardening surfaced in review and fixed in the same change:** (a) The token-aligned prefix match was vulnerable to single-word cache entries acting as false-positive prefixes — e.g. a cache entry titled "The" (the 2017 film, real TMDB id 480530) would prefix-match every release whose name starts with "The" and win when no longer-token candidate exists in the user's cache (most active precisely when the cache is sparse, which is the use case the resolver targets — pre-library-scan grabs). The single-token branch now fail-closes: it requires both a parsed year AND an entry year that matches, instead of inheriting the multi-token branch's fail-open semantics. The multi-token branch keeps fail-open for missing entry-year (legacy entries lack `release_date`/`first_air_date` and the candidate's specificity already protects). (b) `_safe_entry_title` and an `isinstance(date, str)` guard in `_extract_entry_year` defend against malformed cache entries where `title`/`release_date`/`first_air_date` is non-string (dict/list/int) — calling `.strip()` on those raised AttributeError that escaped `_lookup_canonical_in_tmdb` → escaped `_resolve_canonical_title` (its sole try/except wraps `parse_folder_name` only) → propagated out of `_enrich_for_history` to the blackhole main loop and broke history logging for the entire grab event. The per-entry scan loop is now also wrapped in a defensive `try/except Exception: continue` so any future per-entry surprise drops just that entry. (c) Import-failure and cache-load-failure paths now emit `logger.debug` lines so silent grab-event degradation is discoverable in production instead of being swallowed without trace. (d) `_BARE_YEAR_RE` is now a module-level constant instead of being recompiled on every call, matching the pattern of all other regexes in `blackhole.py` and `library.py`. **36 new tests** in `tests/test_blackhole.py:TestEnrichForHistoryRobustParsing` (10 cases) and `TestCanonicalTitleResolution` (26 cases) cover: parens-year, bracket-year, dash-separated year, edition tag stripping, indexer URL prefix, the original Gattaca actor+genre regression, regression guards on every existing parse_release_name and _enrich_for_history input, direct year-qualified cache hit, prefix-match recovery from actor/genre junk, year-mismatch exclusion, longest-prefix tiebreaker, mid-string match rejection (must be true prefix), shows-section vs movies-section routing, empty cache fallback, `_load_cache` exception fallback, `parse_folder_name` exception fallback, empty/None filename pass-through, extension-only filename, missing parsed year (no year filter), multi-token entry-year fail-open, non-dict cache entry skip, end-to-end `_enrich_for_history` cache hit, **the two reviewer-surfaced fixes** (single-token prefix requires year confirmation in three sub-cases: year mismatch, no parsed year, no entry year; single-token year match accepted; non-string title in entry doesn't crash; non-string title during prefix scan skipped without poisoning sibling matches), and direct unit tests for `_extract_entry_year` (release_date branch, first_air_date branch, missing date returns None, malformed/non-string date returns None). Full suite: 1815 tests pass.

### Changed

- **Activity feed: library-scan durations now switch to seconds once they cross 1000ms**: The Activity-feed `task_library_scan` row used to render `Library scan — 565 movies, 117 shows, 80645ms` for a 80.6-second scan, which gets unreadable past a few hundred ms. A new `fmt_duration_ms(dur)` helper in `utils/activity_format.py` formats sub-second values as integer milliseconds (`850ms`) and ≥1000ms values as seconds with one decimal (`1.0s`, `80.6s`). The Python helper and its JS mirror inside `FORMATTER_JS` apply identical rules — both gate on `ms > 0` and reject NaN/non-numeric so the server-rendered `detail` and the live JS renderer never diverge. `utils/scheduled_tasks.py` consumes the same helper for both the legacy `detail=` string and the task runner's `'message'` field, so notifications and logs match the Activity feed. **Hardening surfaced in review and fixed in the same change:** (a) Renamed the helper from `_fmt_duration_ms` (private) to `fmt_duration_ms` and added it to `__all__` since `scheduled_tasks.py` imports it across module boundaries — leading-underscore symbols imported by other modules are a footgun for future refactors. (b) Negative durations (e.g., from clock skew) used to render as `-500ms` in Python but were dropped by the JS guard `ms > 0` — Python now matches by returning `''` for non-positive or NaN inputs, so the same event renders identically in both paths. (c) `scheduled_tasks.py`'s fallback was `_fmt_duration_ms(duration_ms) or f'{duration_ms}ms'`, which would render `'Noneems'` if a future upstream change pushed a non-numeric into `scan_duration_ms`; switched to `or '0ms'` so the fallback is always a sane string. Three new tests in `tests/test_activity_format.py` cover the sub-second/exact-1000ms boundaries and the non-positive/non-numeric drop set; the existing test was updated from asserting `'80645ms'` to `'80.6s'`. Full suite: 1780 tests pass.

### Added

- **Status page: Library Composition card**: Adds a full-width card on the Status dashboard summarizing the on-disk library: total items / movies / shows / episodes / size, plus per-source breakdowns (local, debrid, both) for movies and shows with a stacked color bar and per-bucket counts + sizes. Episode counts and sizes are bucketed by episode-level `source` so a `source='both'` show that's still mostly on debrid is rendered correctly. Powered by a new `compute_library_stats(data)` helper in `utils/library.py` and a non-blocking `LibraryScanner.get_cached_stats()` accessor that reads only the in-memory cache — never triggers a scan on the /api/status poll path. Stats piggyback on the existing /api/status response so the dashboard's auto-refresh already drives it; the card stays hidden until the first scan completes (zero-item library shows nothing). **Concurrency hardening surfaced in review and fixed in the same change:** `_scan_effects._cleanup_disc_rips` does an in-place `movies[:] = [...]` slice-assignment AFTER the scan cache is published to readers (refresh() publishes before running effects), which would race with the new poll-path reader and produce double-counted or skipped items mid-cleanup. `get_cached_stats()` now shallow-copies the top-level `movies`/`shows` lists under `_lock` before releasing it, isolating the iteration in `compute_library_stats` from any subsequent in-place mutation. The outer `to_dict()` exception path also now logs at DEBUG so a future regression in the stats helper surfaces in the journal instead of silently dropping the card. The dashboard `buildBar` segments now `esc()` the count interpolated into `title` attributes for defense-in-depth even though the values are server-trusted ints. 10 new unit tests in `tests/test_library.py:TestComputeLibraryStats` cover empty data, source bucketing, episode-level source distribution, episode source falling back to show source, unknown-source fallback, invalid `size_bytes` handling, scan-metadata passthrough, `get_cached_stats()` returning `None` on empty cache, populated-cache happy path, and the concurrent-mutation isolation guarantee. - **Settings → Backup & Restore: per-row Delete buttons for archives and pre-restore snapshots**: The Recent backups dropdown now shows two sections — **Archives (N)** with the existing scheduled `.tar.gz` files plus a new red Delete button on each row, and **Pre-restore snapshots (N)** which surfaces the `pre-restore-<timestamp>/` (and legacy `<timestamp>/`) directories that `restore_from_blob` writes before overwriting config. - **`.pd_zurg_backup` atomic-swap sidecar renamed to `.zurgarr_backup` with opportunistic migration of stranded legacy files (plan 35 Phase 4)**: The prefer-debrid preference path in `utils/library_prefs.py::replace_local_with_symlinks` performs an atomic three-step swap — rename the local file to a transient sidecar, create a symlink at the original path, delete the sidecar on success (or rename it back on symlink failure). The sidecar filename extension moves from the legacy `.pd_zurg_backup` to `.zurgarr_backup` in 2.19.0. The sidecar lives for milliseconds in the normal-completion path — rename → symlink → remove all happen inside a single function call — so it is almost never observable on disk; the only way a stranded legacy-named sidecar persists is if the container crashed (SIGKILL, host power-loss, kernel panic) in the tiny window between the `os.rename` and the subsequent `os.remove`. The plan's scanner-driven migration prose was overstated — no scanner walks the local library looking for stale backups, because the sidecar wasn't designed to persist. The real migration surface is inside `replace_local_with_symlinks` itself: before creating a fresh backup at a given path, the function now checks for a stranded `.pd_zurg_backup` at the SAME path and — only if no entry of any kind already occupies the `.zurgarr_backup` slot — renames the legacy-named sidecar forward to the new extension, so the rest of the swap logic (including the rollback rename on symlink failure) operates under the new name uniformly. The guard composes three distinct checks, each with a concrete failure mode it eliminates: `os.path.isfile(legacy_backup_path)` rejects a directory or FIFO that happens to share the legacy name (the pre-2.19 code path only ever created regular files, so anything else is hostile or spurious and renaming it forward would break the subsequent `os.rename(real_local, backup_path)`); `not os.path.islink(legacy_backup_path)` rejects a symlink at the legacy path (renaming a symlink would move the REFERENCE forward, then the atomic swap would replace the symlink itself with the user's local file and orphan whatever the symlink used to point at); `not os.path.lexists(backup_path)` rejects a pre-existing new-name entry of ANY kind — including a dangling symlink, which the naive `os.path.exists` check would report as missing and which would then be silently replaced by the rename. An `OSError` from the rename (EACCES on a read-only mount, EXDEV across filesystems, etc.) is swallowed at `DEBUG` level via `logger.debug` — the legacy file stays put, the next swap attempt on the same path retries the migration, and a persistent permission problem surfaces in the log rather than looping silently forever. No `warn_once` deprecation on this surface: the migration is file-system level, happens in a crash-recovery window the user can't meaningfully act on, and each stranded sidecar migrates exactly once when the user next triggers a prefer-debrid swap on that specific file. Seven new tests in `tests/test_library_prefs.py::TestReplaceLocalWithSymlinks` (the function had zero test coverage before Phase 4) lock down: the canonical clean-swap with the new extension (local file becomes a symlink to the translated `BLACKHOLE_SYMLINK_TARGET_BASE`-rooted path, no sidecar under either extension remains); the stranded-legacy forward-migration (pre-seeded `.pd_zurg_backup` is gone after a successful swap — renamed forward to `.zurgarr_backup`, then consumed and deleted as the swap's own transient backup); the migration's guard against clobbering a pre-existing regular-file new-name backup; the failure-path rollback (a monkeypatched `os.symlink` that raises causes the original file to be restored with byte-identical contents, no sidecar remains under either extension, and the error message to the caller starts with `"Symlink failed (restored):"` so the UI can distinguish recovered-from failures from hard failures); and three hardened-guard cases — legacy sidecar that is a DIRECTORY (untouched, along with any squatter contents inside it), legacy sidecar that is a SYMLINK to a regular file elsewhere (untouched, symlink target unmodified), and a DANGLING new-name symlink (guard sees it via `lexists`, legacy file-name sidecar is preserved rather than silently replacing the dangling link). **At 2.20.0** the migration pass is removed and the function reads/writes only the new extension; any `.pd_zurg_backup` files that haven't been encountered by a swap on their specific path by then stay on disk as orphaned artifacts the user must clean up manually — negligible in practice because they exist at all only if a crash stranded them in the sub-second swap window.

- **Direct WebDAV scanning**: Library scanner now queries Zurg's WebDAV API directly via PROPFIND, bypassing the FUSE/rclone mount for directory enumeration. Reduces mount scan time from 10-20 seconds to under 1 second. Falls back to FUSE scanning automatically if Zurg is unreachable.
- **Smart refresh polling**: Library refresh button now polls the backend until the scan completes, replacing the fixed 3-second delay. Shows a scanning indicator and updates the UI automatically when data is ready. Includes error feedback for timeouts and server failures.

### Changed

- **Split scan architecture**: Library scan is now split into a fast read-only phase (mount enumeration, cross-referencing, TMDB enrichment) and a background effects phase (preference enforcement, arr searches, symlink creation). The UI receives fresh library data after the read phase completes (~1-5s) while effects continue running in the background (~30-60s).
- **Season pack preference**: Scanner now prefers season pack files over individual episode downloads, using per-season episode count to avoid lower-quality mega-packs beating higher-quality season packs.
- **Fresh preferences**: `/api/library` response always returns preferences fresh from disk instead of using stale scan-time values.
- **Concurrent scan protection**: Added `_effects_running` guard to prevent overlapping preference enforcement and arr API calls from concurrent refresh requests.
- **Cache safety**: API responses now use a shallow copy of cached scan data to prevent mutation of the shared cache object across concurrent requests.
- **Stale cache during scan**: `get_data()` returns stale cached data when a background scan is running instead of triggering a duplicate synchronous scan.
- **Grid re-render optimization**: Smart poll skips full grid re-render to prevent poster image flicker during background data updates.

### Fixed

- **Indexer routing**: Debrid tag now applied to torrent indexers, and re-search triggered for missing content after routing changes.
- **Library scanner rescans**: Fixed rescan triggers and improved symlink target path validation to reject paths outside the mount.
- **False 'both' source**: Debrid symlinks in local library scan no longer cause false 'both' source classification. Expanded `verify_symlinks` to check local library directories.
- **Title parsing**: Filter non-media folders (plex versions, subs, featurettes), fix empty parentheses in titles, and prevent MAX title corruption from quality pattern over-matching.
- **RD torrent monitoring**: Fixed monitor polling deleted RD torrents and accept `selectFiles` 202 status responses.
- **Alt-retry race condition**: Hardened failure paths in automatic alternative release retry logic.
- **Episode dedup regex**: Updated to handle Sonarr-standard naming patterns (S01E01 format).
- **Download routing e2e**: Fixed usenet client skipping, indexer downloadClientId override clearing, and stale unavailable queue item cleanup.
- **Stale pending entries**: Auto-clear pending entries for titles removed from the library.
- **TMDB year-filter search failure**: TMDB searches with a year filter now retry without the year when the filtered search returns no results. Fixes shows like "Marvel's Spidey and His Amazing Friends" where the folder year (season air date) doesn't match the show's premiere year, preventing TMDB caching and alias-based dedup.
- **WebDAV scanner Zurg compatibility**: WebDAV PROPFIND scanner now handles both absolute (`/dav/movies/...`) and relative (`folder/file`) hrefs from Zurg. Also detects when Zurg doesn't support recursive depth-infinity PROPFIND (returns folders without files) and falls back to FUSE mount scanning automatically.
- **Stale debrid symlink cleanup**: `verify_symlinks` now checks symlinks pointing to `BLACKHOLE_SYMLINK_TARGET_BASE` (e.g. `/mnt/debrid/`) in addition to the rclone mount path. Previously only checked the mount path, so symlinks created with a different target base were never cleaned up when torrents expired.
- **Faster debrid library on startup**: When the rclone mount appears after the initial scan has already started, the scanner now automatically triggers a follow-up scan so debrid content appears within seconds instead of waiting for the next scheduled scan (up to 2 minutes).
- **Title parsing for mid-string year**: Folder names like `Movie (2000) DC (1080p BluRay...)` now correctly extract the year before quality truncation, preventing mangled titles like `Movie (2000) DC (1080p` and duplicate library entries without posters.
- **Symlink creation guard for empty local library**: Debrid symlink creation now skips when the local library scan found zero local content, preventing symlink pollution when network mounts (NFS/SMB) aren't propagated into the container.
- **TMDB year-preference matching**: TMDB search now prefers results whose release year matches the folder year instead of blindly taking the first (most popular) result. Only activates when the top result is >2 years off (avoiding false corrections from season air years), limits scan to top 5 results, and skips year preference entirely on fallback-no-year retries. Fixes movies like "Cover Up (2025)" incorrectly showing the poster and metadata for the 1983 French film "La Crime".
- **Verify symlinks deleting valid debrid symlinks**: `verify_symlinks` and blackhole's `_cleanup_symlinks` were checking symlink targets against `BLACKHOLE_SYMLINK_TARGET_BASE` (e.g. `/mnt/debrid`) which only exists inside Radarr/Sonarr's container, not pd_zurg's. Every debrid symlink appeared broken and was removed. Now translates target paths to the rclone mount before checking existence. Also adds a mount health check (aborts if rclone mount is unresponsive), a safety threshold (refuses to delete >50 symlinks if >50% appear broken), and cleans up empty parent directories after deletion so the library scanner doesn't misclassify them as local content.
- **Phantom local movie classification**: After symlinks were deleted, movie folders containing only Radarr metadata (.nfo, .jpg) but no media files were misclassified as `source='local'`, permanently blocking symlink recreation. `_scan_local_movies` now requires at least one media file to classify a directory as local. Same fix applied to shows via `_scan_local_shows`.
- **Symlink dir name mismatch with Radarr**: Debrid symlinks were created under the torrent-parsed title (e.g. `F1 The Movie (2025)`) instead of Radarr's canonical folder name (`F1 (2025)`). Added TMDB ID-based fallback matching when title lookup against Radarr fails, so the symlink lands in the correct Radarr-managed folder. Also fixed `_norm_for_matching` to treat hyphens as word separators ("Cover-Up" matches "Cover Up") and `&` as "and" ("Me, Myself & Irene" matches "Me Myself And Irene"). Movies with `source='both'` now also get symlinks created in Radarr's folder when it's empty, fixing the case where a wrong-named symlink dir existed but Radarr's canonical dir was empty.
